This page covers nested dictionaries.

Follow one key or index at a time and name what each step returns.

Python data structures · visual revision guide

Follow the keys.

Nested dictionaries organise detailed information in layers. Each key opens one named part of the structure; a list inside it may then require a numbered index.

2selector types: key and index

01

Start with the real job

Keep related information together

A user profile may contain contact details, skills and preferences. Nesting keeps each group inside the person it belongs to without flattening everything into unrelated variables.

Outer dictionary

The complete person

Keys such as "name" and "details" divide the profile into named parts.

Nested values

Groups inside groups

A value may itself be a dictionary or list, allowing another layer of organised information.

The key question at every step: what type of value did this key return—a string, list or another dictionary?

02

See the structure before reading it

A dictionary can be a value

users = { "user1": { "name": "Morgan", "age": 25 }, "user2": { "name": "José", "age": 13 } }
STARTusersThe complete outer dictionary.
OUTER KEY["user2"]Returns José’s whole inner dictionary.
INNER KEY["name"]Returns the name stored inside it.
RESULT"José"The requested string value.

03

One named step at a time

Chain keys to retrieve a value

print(users["user2"]["name"])
Printed outputJosé

Plain-English reading: go to "user2" in users, then retrieve the value under "name" in that returned dictionary.

04

Let the returned type choose the next selector

Keys can lead to a list

profile = { "name": "Morgan", "details": { "age": 25, "skills": ["Python", "SQL", "Excel"] } } print(profile["details"]["skills"][0])
Printed outputPython
DICTIONARYprofileBegin with the complete profile.
KEY["details"]Returns a dictionary.
KEY["skills"]Returns a list.
INDEX[0]Returns the first list item: "Python".

Use a key for a dictionary and an index for a list. The selector changes because the type of collection changed.

05

Address first, assignment second

Change an existing nested value

For the next examples, the profile gains a "contacts" dictionary. This is the complete state before the email changes:

profile = { "name": "Morgan", "details": { "age": 25, "skills": ["Python", "SQL", "Excel"] }, "contacts": { "email": "Jamie@example.com", "phone": ["123-456-7890", "987-654-3210"] } }

Before · contacts

Old email

"Jamie@example.com"
profile["contacts"]["email"] = "newJamie@example.com"

After · contacts

New email

"newJamie@example.com"
Resulting contacts dictionary{'email': 'newJamie@example.com', 'phone': ['123-456-7890', '987-654-3210']}

Left-to-right intent: enter "contacts", go to "email", and give that location the new string value.

One more fact: how the line runs

Your left-to-right reading is still the right way to understand what this line is for: go to the key on the left and give it the value on the right. When Python runs the line, it adds one more step: it first works out the value on the right, then stores that result under the label on the left. This matters when the right side is a calculation instead of a plain value.

profile["details"]["age"] = profile["details"]["age"] + 1
Right side works out to26Resulting dictionary{'age': 26, 'skills': ['Python', 'SQL', 'Excel']}

06

A new key creates a new entry

Add data at the correct layer

profile["contacts"]["address"] = "42 Wonderland Ave" print(profile["contacts"])
Printed output{'email': 'newJamie@example.com', 'phone': ['123-456-7890', '987-654-3210'], 'address': '42 Wonderland Ave'}

Why did this add instead of replace? The "address" key did not already exist inside "contacts". Assignment creates it at that layer.

07

Apply the LMS user-profile scenario

Build preferences inside details

Starting structure

A user with details

The nested "details" dictionary contains an email and a "phone" list holding one number.

Your four jobs

  1. Append "555-123-4567" to the phone list.
  2. Create a nested "preferences" dictionary.
  3. Start with theme "light" and notifications True.
  4. Change the theme to "dark" and print user.
Check the complete user-profile exercise
user = { "username": "techlearner101", "details": { "email": "learner@example.com", "phone": ["555-999-0000"] } } user["details"]["phone"].append("555-123-4567") user["details"]["preferences"] = { "theme": "light", "notifications": True } user["details"]["preferences"]["theme"] = "dark" print(user)
Printed output{'username': 'techlearner101', 'details': {'email': 'learner@example.com', 'phone': ['555-999-0000', '555-123-4567'], 'preferences': {'theme': 'dark', 'notifications': True}}}

08

Diagnose the route

Common nested-data traps

Stopped one layer early

profile["details"]Actual: the complete details dictionary

Add another key only when you need one value inside it.

Used an index on a dictionary

profile["details"][0]Actual: KeyError: 0

"details" returns a dictionary, so use one of its keys.

Used a key on a list

profile["details"]["skills"]["Python"]Actual: TypeError

"skills" returns a list, so choose an item with a numeric index.

Added the key at the wrong level

profile["address"] = "42 Wonderland Ave"Actual: address appears beside contacts

Use profile["contacts"]["address"] when the address belongs inside contacts.

09

Before moving on

Quick recall

  1. What makes a dictionary nested?
  2. What does users["user2"] return?
  3. Why does profile["details"]["skills"][0] end with an index?
  4. How does assignment decide whether to add or replace a nested key?
Reveal the quick-recall answers
  1. One of its values is another dictionary or collection.
  2. The complete dictionary stored under "user2".
  3. "skills" returns a list, and lists use numeric indices.
  4. An existing key is replaced; a new key is added at the addressed dictionary layer.

The durable mental model: follow the data one layer at a time. The value returned by the current step tells you whether the next selector must be a key or an index.
